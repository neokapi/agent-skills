# Add i18n to a project

kapi's goal for i18n is **zero toil**: after adoption, a developer ships
localized features by writing plain source-language text, and translation is an
ambient consequence of a source change — extraction, sync, QA, and fill happen
in the build and CI, never as a chore someone remembers. Every framework this
skill knows is graded on how close it gets (the **Toil Index**, below); your
job is to (1) detect the stack, (2) install the *recommended configuration*
that earns the framework its grade — not the stock setup — and (3) warn the
user honestly when their stack can't get below T3.

## Step 1 — detect the stack

Consult [i18n/frameworks.yaml](i18n/frameworks.yaml) — the registry of every
framework this skill knows: detection signals (`detect.deps` in
package.json/pubspec/Gemfile/go.mod/csproj, `detect.files` markers), catalog
layout, the kapi preset and flags to use, toil scores, and status
(`recommended`/`supported`/`caution`/`avoid`). Check dependency manifests and
marker files against it. `kapi init --list-presets` shows the presets kapi can
scaffold directly.

## The Toil Index (how to read the grades)

Every framework carries five axis scores 0–3 (**W** marking, **X** extraction,
**S** sync, **R** runtime-correctness, **E** ecosystem risk) and a grade
derived from them, always quoted *in the recommended configuration*:

- **T0 — add and forget.** Plain source text is enough; everything else is
  build/CI. (kapi-react, iOS String Catalogs)
- **T1 — one habit.** One step per feature — wrap with source text or run an
  extract; tooling catches the rest. (gettext, Angular+merge-tool, Paraglide)
- **T2 — standing chore.** Someone owns key hygiene and sync; i18n appears in
  sprint planning. (react-i18next, vue-i18n, Flutter, Android)
- **T3 — recurring project.** Releases gate on manual passes; drift reaches
  prod. **Tell the user this before adopting**, and name the nearest
  lower-toil alternative.
- **T4 — you're on your own.** Hand-edited catalogs, no validation. Warn
  explicitly; offer the migration path.

When you install the tools listed in the registry's `buys` field, say so —
that delta (e.g. Angular: T3 stock → T1 with `ng-extract-i18n-merge`) is the
justification for each setup step. If the user declines the recommended
config, quote the stock grade instead and say what they are keeping.

## Rules that apply to every stack

- **Key model: follow the stack's idiom.** kapi's own stack (kapi-react) uses
  the English source text as the key — never invent message IDs there. But
  when plugging into an ID-based stack (next-intl, i18next, Android, Rails),
  keep its idiom: converting a framework against its native key model is a
  reliable toil generator. The per-ecosystem file states the idiom.
- **Readiness first.** `kapi pseudo-translate <catalog> --target-lang qps`
  before any real translation — it surfaces hardcoded/missed strings and is
  the pseudo-localization pass most stacks lack (this is what buys the R-axis
  improvement).
- **Route translation through kapi.** Translate the catalogs yourself but via
  kapi (`kapi translate <catalog> --target-lang <l>`, or extract → translate →
  merge) so terminology, placeholders, and format survive — never hand-edit
  target catalogs. Structure, keys, and placeholders (`{{name}}`, `{0}`, ICU,
  printf, `%@`) are preserved; only values translate. A provider credential is
  needed only for unattended runs.
- **Repeatable runs get a project.** For CI / recurring locales, scaffold with
  `kapi init --framework <preset> --target-locale <l> …` (preset ids and extra
  `--format` flags are in the registry). One-off file translations need no
  project.
- **The gate is rendering, not the catalog.** After wiring, run the app in the
  target locale and confirm the visible UI is translated. If it still shows
  source language the wiring is wrong, not the translation — check extraction,
  client/server component boundaries, and dictionary load timing per the
  ecosystem file. In a project, `kapi check --ship` must pass before you call it
  done.
- **Bind brand + glossary** so translations stay on-brand and consistent (see
  [localize.md](localize.md)).
